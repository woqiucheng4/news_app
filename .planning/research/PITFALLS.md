# Domain Pitfalls: AI-Powered Information Aggregation

**Domain:** AI 驱动的信息聚合应用
**Researched:** 2026-05-08
**Confidence:** MEDIUM (基于行业案例和已知模式，部分需实际验证)

---

## Critical Pitfalls

### Pitfall 1: AI API 成本失控

**What goes wrong:**
每篇文章都调用 AI API 生成摘要，随着订阅源增多和更新频率提高，API 费用指数级增长。月预算 $50 在几天内耗尽。典型场景：100 个订阅源，每小时更新，每篇文章 500 token 输入 + 200 token 输出，仅摘要一项月费就可能超过 $200-500。

**Why it happens:**
开发者在测试阶段用少量数据验证功能，没有估算规模化后的成本。忽略了"免费层"和"实际生产流量"之间的巨大差距。更致命的是：相同或相似内容被重复摘要（去重失败导致重复调用）。

**How to avoid:**
- 实施严格的 token 预算系统：每用户每日/每月 token 上限
- 使用语义缓存（semantic caching）：相似内容命中缓存则不调用 API
- 分层模型策略：绝大多数摘要用 Claude Haiku/GPT-4o-mini，仅付费用户的深度分析用大模型
- 先去重、再摘要：确保同一事件只被摘要一次
- 实时监控 API 调用量和成本，设置自动熔断阈值（如日预算达到 80% 时降级）
- 利用 Anthropic/OpenAI 的 prompt caching 功能（缓存相同 system prompt，可节省 90% 输入 token 费用）

**Warning signs:**
- 开发阶段 API 费用已超过 $10/月
- 相似文章被多次摘要（去重率低于 70%）
- 没有成本监控 dashboard
- system prompt 每次都完整发送（未利用 prompt caching）

**Phase to address:**
Phase 1（爬虫+摘要核心管道）——必须在第一阶段就建立成本控制机制，不能事后补救。

### Pitfall 2: 内容去重失败导致信息轰炸

**What goes wrong:
同一新闻事件被 10 个不同媒体报道，用户看到 10 条几乎相同的摘要。更糟的是：每条都消耗了 AI API 调用。用户体验急剧下降，感觉是"垃圾信息聚合器"而非"智能摘要"。

**Why it happens:**
新闻行业的特殊性——通讯社（AP、Reuters、AFP）供稿被大量媒体转载，标题和正文高度相似。简单用 URL 去重无效（不同 URL 指向相同内容）。用标题精确匹配也不够（"Apple 发布新 iPhone" vs "Apple unveils new iPhone" 被视为不同文章）。

**How to avoid:**
- 实施多层去重策略：
  1. URL 去重（第一层，最快）
  2. 标题相似度（MinHash/编辑距离，第二层）
  3. 内容指纹（SimHash/MinHash shingling，第三层）
  4. 语义相似度（embedding 向量余弦相似度 > 0.85，第四层）
- 建立"事件聚类"（event clustering）：将同一事件的多篇报道聚合为一个事件卡片
- 优先使用 RSS——RSS 源通常已经是精选内容，重复率低于爬虫

**Warning signs:**
- 信息流中超过 30% 的内容看起来高度相似
- 用户反馈"看到太多重复内容"
- 去重只用了 URL 级别匹配
- 没有事件聚类机制

**Phase to address:**
Phase 1（爬虫+摘要核心管道）——去重是管道的核心组件，不是优化项。

### Pitfall 3: 爬虫被反爬措施全面封锁

**What goes wrong:**
主流新闻网站使用 Cloudflare Bot Management、PerimeterX（现 HUMAN Security）、Akamai Bot Manager 等反爬方案。爬虫在部署后几天内被全面封锁——IP 被封、返回 CAPTCHA、甚至返回虚假内容。整个数据采集管道瘫痪。

**Why it happens:**
开发者在本地测试时用少量请求验证爬虫逻辑，没有考虑生产环境的规模化请求。忽略了现代反爬系统的行为分析能力：不仅检测 User-Agent，还分析鼠标轨迹、JavaScript 执行环境、请求频率模式、TLS 指纹等。

**How to avoid:**
- RSS 优先策略（项目已采用）：RSS 是网站主动提供的数据接口，不存在反爬问题
- 对于无 RSS 的源，使用官方 API（如 Twitter API、Reddit API）而非爬虫
- 如果必须爬虫：
  - 使用代理轮换（但增加成本）
  - 尊重 robots.txt 和 Crawl-delay
  - 设置合理的请求间隔（至少 5-10 秒/请求）
  - 使用真实的浏览器指纹（Playwright 而非 requests）
  - 清晰的 User-Agent 标识
- 接受现实：不可能爬取所有网站，专注于 RSS + API 可覆盖的源

**Warning signs:**
- 爬虫频繁收到 403/429 状态码
- 需要越来越复杂的反检测措施
- 爬虫代码中出现大量"绕过"逻辑
- 维护爬虫的时间超过开发核心功能的时间

**Phase to address:**
Phase 1（爬虫+摘要核心管道）——在架构设计阶段就决定"RSS + API 优先，爬虫兜底"的策略。

### Pitfall 4: RSS 源可靠性衰减

**What goes wrong:**
RSS feed URL 静默变更（无重定向）、feed 内容格式不规范导致解析失败、网站停止维护 RSS 输出、feed 更新频率不可预测。3-6 个月后，大量订阅源失效，用户发现"我的订阅已经一周没更新了"。

**Why it happens:**
RSS 在很多网站是"遗留功能"，不被重视。网站改版时 RSS 端点经常被悄悄移除或变更。开发者假设 RSS 是稳定的基础设施，实际上它是"尽力而为"的服务。

**How to avoid:**
- 实施 feed 健康监控：追踪每个 feed 的最后成功抓取时间、HTTP 状态码、内容变化
- 自动降级：当 RSS 失效时，自动尝试备用方案（如该源的其他 feed URL、或该网站的 sitemap.xml）
- 告警机制：feed 连续失败 N 次后通知开发者
- feed 验证：定期使用 W3C Feed Validation Service 检查 feed 格式
- 缓存 feed 内容：当源暂时不可用时，仍可展示缓存的内容
- 维护一个"已知好用的 feed"数据库，包括备用 URL

**Warning signs:**
- 没有 feed 健康监控系统
- 不知道哪些 feed 已经失效
- 用户投诉"订阅不更新"才发现问题
- 没有 feed 失效的自动处理逻辑

**Phase to address:**
Phase 1（爬虫+摘要核心管道）——feed 监控是管道的必备组件。

### Pitfall 5: 版权纠纷与法律风险

**What goes wrong:**
新闻出版商投诉或起诉：展示过多原文内容（超出"合理使用"范围）、AI 摘要"过于详细"等同于复制、欧盟版权指令（Article 15）的"链接税"要求。更严重的是收到 DMCA Takedown 通知或被应用商店下架。

**Why it happens:**
开发者认为"摘要+链接"就万无一失，但法律边界模糊。关键案例：AP v. Meltwater（2013）——即使是摘要+链接，法院也判定非合理使用。AI 生成的摘要可能比人工摘要更详细，更容易触碰版权红线。

**How to avoid:**
- 严格限制摘要长度：每篇摘要不超过 2-3 句话，仅概括核心事实
- 不复制原文句子：AI prompt 必须明确要求"用自己的话重新表述"
- 展示来源归属：每条摘要必须包含原文链接和来源名称
- 不缓存全文（项目已采用）：仅存储标题、摘要、URL
- 建立 DMCA 响应流程：收到投诉后快速响应
- 监控欧盟版权指令的实施情况（如需在欧盟运营）
- 考虑与主要出版商建立合作关系

**Warning signs:**
- AI 摘要包含原文的完整句子
- 摘要长度超过 100 字
- 没有 DMCA 响应流程
- 不清楚目标市场的版权法律要求

**Phase to address:**
Phase 1（爬虫+摘要核心管道）——摘要生成的 prompt 设计必须在第一阶段就考虑版权合规。

### Pitfall 6: 推送通知疲劳导致用户流失

**What goes wrong:**
用户订阅 5 个热门话题，每天收到 50+ 条推送通知。用户在 3 天内关闭通知权限，1 周内卸载应用。推送通知从"有价值的功能"变成"骚扰工具"。

**Why it happens:**
项目设计中"话题重大更新"和"突发事件"的定义过于宽泛。热门话题（如科技新闻）每天有数十条"重大更新"。没有频率控制机制，每条符合条件的新闻都触发推送。

**How to avoid:**
- 实施推送频率上限：每用户每天最多 N 条推送（建议默认 3-5 条）
- 智能聚合推送：将同一话题的多条更新合并为一条推送（"您关注的 AI 话题今日有 5 条重要更新"）
- 分级推送策略：
  - 紧急/突发：立即推送（但定义严格，如重大灾难、市场崩盘）
  - 重要更新：每日汇总推送（定时，如早 8 点）
  - 一般更新：仅在应用内展示
- 新用户引导：默认关闭推送，引导用户按需开启
- 推送效果监控：追踪推送打开率、通知关闭率、卸载率

**Warning signs:**
- 每用户日均推送超过 10 条
- 推送打开率低于 5%
- 大量用户在安装后 3 天内关闭通知权限
- 没有推送频率控制逻辑

**Phase to address:**
Phase 2（用户系统+推送）——推送策略必须在实现推送功能时就设计好，不能先推送再优化。

### Pitfall 7: GDPR 合规疏忽

**What goes wrong:**
在欧盟运营但没有合规的数据处理流程：爬取了包含个人数据的内容（如社交媒体帖子中的用户名）、用户数据存储不符合数据最小化原则、没有数据删除机制、隐私政策不完整。被数据保护机构调查或罚款。

**Why it happens:**
GDPR 合规被认为是"法律问题"而非"工程问题"，被推迟到"以后再做"。实际上 GDPR 要求"隐私设计"（Privacy by Design），必须从架构层面嵌入。

**How to avoid:**
- 数据最小化：只收集提供服务所必需的数据
- 爬虫过滤：自动过滤掉个人数据（用户名、头像、个人评论）
- 用户数据管理：提供数据导出和删除功能（Right to Erasure）
- 数据处理协议（DPA）：与所有第三方服务（AI API、推送服务、数据库托管）签署
- 隐私政策：明确说明数据收集范围、用途、存储位置
- 数据存储位置：使用欧洲区域的服务器（如 AWS eu-west-1）
- 实施 Consent Management：用户明确同意数据处理

**Warning signs:**
- 爬虫存储了社交媒体用户的个人信息
- 没有用户数据删除功能
- 不清楚第三方服务的数据处理条款
- 隐私政策是模板化的，不反映实际数据处理

**Phase to address:**
Phase 1（架构设计）——数据模型必须考虑 GDPR 要求。Phase 2（用户系统）实现具体合规功能。

### Pitfall 8: 应用商店审核被拒

**What goes wrong:**
Apple App Store 以"功能不足"（Guideline 4.2）或"垃圾应用"（Guideline 4.2.6）为由拒绝上架。Google Play 以"重复内容"或"内容政策违规"为由下架。审核周期长达数周，延误发布时间线。

**Why it happens:**
Apple 对新闻聚合类应用审核严格——如果应用只是简单展示 RSS 内容而没有"足够的独特价值"，会被判定为"垃圾应用"。AI 摘要功能可能被视为"不够独特"。此外，如果展示的内容包含敏感信息（如暴力、色情），可能触发内容审核问题。

**How to avoid:**
- 确保应用有足够的独特功能：AI 摘要、个性化推荐、话题聚类——而不仅仅是 RSS 阅读器
- 准备详细的 App Store 描述，突出 AI 价值
- 实施内容过滤：自动过滤敏感/违规内容
- 提前准备审核材料：截图、演示视频、功能说明
- 遵循 Apple Human Interface Guidelines 和 Google Material Design
- 先提交 TestFlight/内测版本，收集反馈后再正式上架
- 预留 2-4 周的审核缓冲时间

**Warning signs:**
- 应用功能与现有 RSS 阅读器高度相似
- 没有内容过滤机制
- 不熟悉 Apple/Google 的审核指南
- 没有为审核准备专门的演示内容

**Phase to address:**
Phase 3（发布准备）——但功能设计阶段就应考虑审核要求。

### Pitfall 9: 数据库设计无法支撑信息聚合的特殊需求

**What goes wrong:**
信息聚合应用有独特的数据特征：大量重复/相似内容、时间序列数据（新闻流）、非结构化元数据（AI 生成的标签、聚类信息）。使用通用的数据库 schema 设计，导致：查询性能差（信息流加载慢）、去重效率低、难以实现事件聚类。

**Why it happens:**
开发者使用通用的 CRUD 应用模式设计数据库，没有考虑信息聚合的特殊查询模式：按时间排序的信息流、按相似度去重、按事件聚类、按用户偏好推荐。

**How to avoid:**
- 设计专用的数据模型：
  - `articles` 表：存储原始文章元数据
  - `events` 表：事件聚类（一个事件对应多篇文章）
  - `summaries` 表：AI 生成的摘要（与事件关联，非文章）
  - `user_subscriptions` 表：用户订阅的话题/源
- 使用 JSONB 存储非结构化数据（项目已采用 PostgreSQL + JSONB）
- 为时间序列查询建立索引（created_at, published_at）
- 考虑使用 pgvector 扩展支持向量相似度搜索（用于语义去重）
- 设计数据保留策略：旧文章自动归档/删除

**Warning signs:**
- 信息流查询需要多表 JOIN
- 没有事件聚类的数据结构
- 去重逻辑在应用层而非数据库层
- 没有数据保留/清理策略

**Phase to address:**
Phase 1（核心数据模型）——数据库 schema 设计必须在开发初期就考虑信息聚合的特殊需求。

### Pitfall 10: 免费增值模式的转化陷阱

**What goes wrong:**
免费用户用量远超预期（大量用户注册后高频使用免费功能），但付费转化率极低（<1%）。免费层的 AI 摘要成本由开发者承担，入不敷出。更糟的是：免费层限制太严导致用户流失，限制太松导致无付费动力。

**Why it happens:**
在 $50/月预算下，免费用户的 AI API 调用成本很容易超出预算。"有限订阅数"和"每日查看限制"的具体数值很难平衡——太低用户不来，太高不赚钱。

**How to avoid:**
- 精确计算免费用户的成本：每个免费用户每月的 AI API 成本是多少？
- 设置免费层的硬性成本上限：免费用户共享一个 token 预算池
- 免费层功能差异化：
  - 免费：基础摘要（短、简略）、有限订阅数（3-5）、有广告/品牌水印
  - 付费：深度摘要、无限订阅、无广告、优先推送
- 实施"试用期"策略：新用户 7 天免费体验付费功能
- 监控关键指标：免费用户成本、付费转化率、用户生命周期价值（LTV）

**Warning signs:**
- 免费用户数量增长但付费用户停滞
- 免费用户的 AI API 成本超过总收入
- 不清楚每个免费用户的实际成本
- 免费层和付费层的差异不明显

**Phase to address:**
Phase 2（用户系统+商业模式）——定价策略必须在用户系统设计时就确定。

### Pitfall 11: 热点数据采集的法律合规风险

**What goes wrong:**
从微博、知乎、百度等平台爬取热搜/热榜数据用于商业产品，被平台以违反《反不正当竞争法》起诉，或因突破技术防护被以《刑法》285条追诉。大众点评诉百度案（判赔 323 万）是典型先例——将爬取的平台数据在自己产品中展示，构成"实质性替代"。

**Why it happens:**
开发者认为"公开数据就可以随便爬"，忽略了三个关键边界：
1. robots.txt 协议的法律效力（虽然学术界有争议，但实务中仍被法院参考）
2. 平台 ToS（用户协议）中明确禁止数据爬取/商用的条款
3. 展示方式——如果爬取的数据在自己产品中构成对原平台的"实质性替代"，即使数据是公开的也可能构成不正当竞争

**How to avoid:**
- **数据源策略：只接入非高风险源，不碰高风险源**
  - 接入的源（公开数据 + 无强反爬 + 无严格 ToS 限制）：Hacker News API、Reddit API、GitHub Trending、Google Trends、Product Hunt、微博热搜、知乎热榜、百度热搜、B站热门
  - 明确排除的源（高风险，不接入）：Twitter/X（API 付费 + ToS 限制严格）、抖音（反爬极严 + 法律风险不确定）
  - 一次性接入全部非高风险源，不做分批
- **技术合规清单（每个源必须遵守）：**
  1. 检查并尊重 robots.txt，禁止的路径不爬
  2. 控制频率：每个源 15-30 分钟一次，单次请求间隔 ≥ 3 秒
  3. User-Agent 标识产品名称和联系方式（如 `NewsFlow/1.0 (+https://newsflow.app/bot)`）
  4. 不突破任何技术防护（验证码、429 限流、封 IP 立即停止）
  5. 保留完整访问日志（时间、目标 URL、HTTP 响应码），作为合规证据
- **数据使用合规清单：**
  1. 只取元数据：热搜标题、排名、热度值，不爬评论正文
  2. 不存储个人信息：如果热搜包含人名，只取话题标签，不关联用户
  3. AI 摘要为原创生成，不复制原文
  4. 每条热点标注来源归属（"来源：微博/知乎/..."）
  5. 点击跳转到原始平台，不做站内替代阅读
  6. 热点数据只保留 7-30 天，不做永久存储
- **响应机制：**
  1. 收到任何平台的 DMCA/律师函/停止要求，48 小时内下架对应源
  2. 提供平台退出机制：平台可要求从产品中移除其数据
  3. 建立法律风险评估文档，定期审查数据源合规状态

**法律依据（中国）：**
- 最高检（2025.11）观点："数据爬虫行为只要并未超出数据来源主体初次同意的授权范围，应视为存在拟制同意"——公开热搜数据属于用户首次授权公开的范围
- 趋势：从"合约权利标准"（看 robots.txt/ToS）转向"技术障碍标准"（看是否突破技术防护）——公开数据 + 不突破技术防护 = 合规风险可控
- 但《反不正当竞争法》仍可能适用——关键是不做"实质性替代"，只展示摘要+链接

**Warning signs:**
- 爬取了需要登录才能访问的数据
- 未检查 robots.txt 就开始爬取
- 在产品中原样展示平台数据（未做 AI 摘要转换）
- 收到平台停止要求后未响应
- 热点数据中包含用户个人信息
- 没有访问日志记录
- 接入了明确排除的高风险源（Twitter/X、抖音）

**Phase to address:**
Phase 1（热点功能实现时）——合规策略必须在采集管道设计阶段就嵌入，不能事后补救。

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| 用正则而非 AI 做内容过滤 | 零成本 | 高误判率，漏掉违规内容 | 仅在 MVP 初期，必须尽快替换 |
| 每次都完整发送 system prompt | 实现简单 | 浪费 token，增加成本和延迟 | 仅在开发测试阶段 |
| 用 cron job 而非消息队列处理爬取任务 | 实现简单 | 无法处理任务失败、无重试机制、无法扩展 | 仅在用户数 < 100 的初期 |
| 硬编码爬虫配置（URL、选择器） | 快速开发 | 每个网站改版都需要代码修改 | 绝对不可接受 |
| 不做 feed 健康监控 | 少写代码 | feed 失效后无法及时发现 | 绝对不可接受 |
| 用 SHA256 做内容去重 | 实现快 | 无法处理近似重复（改一个字就不同） | 仅在初期，需尽快升级为 SimHash/MinHash |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| AI API (Claude/GPT) | 不处理 rate limit 和超时 | 实施指数退避重试、fallback 到备用模型、设置超时 |
| Firebase Cloud Messaging | 推送所有更新给所有用户 | 实施用户偏好过滤、频率控制、静默时段 |
| RSS Parser | 不处理格式不规范的 feed | 使用容错性强的解析器（如 feedparser）、处理编码问题 |
| PostgreSQL | 不考虑 JSONB 查询性能 | 为常用查询路径建立 GIN 索引、避免过度使用 JSONB |
| 社交媒体 API (Twitter/Reddit) | 不了解 API 限制和定价变化 | 定期检查 API 定价变更、准备降级方案 |
| 代理服务 | 使用免费/低质量代理 | 使用可靠的付费代理服务、监控代理质量 |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| 信息流查询未优化 | 首页加载 > 2 秒 | 为 user_id + published_at 建立复合索引、使用 cursor-based pagination | 订阅数 > 50 且文章数 > 10K |
| 去重在应用层逐篇比对 | 爬取处理时间线性增长 | 使用数据库级别的去重（唯一约束 + 语义指纹列） | 每日新增文章 > 1K |
| AI 摘要同步处理 | 爬取完成后长时间等待摘要生成 | 使用异步任务队列（Celery）并行生成摘要 | 单次爬取 > 100 篇文章 |
| 无缓存层 | 数据库 CPU 持续高位 | 在 API 层和数据库层之间加 Redis 缓存 | 并发用户 > 100 |
| 全量推送 | 推送服务响应慢、用户投诉 | 使用批量推送 API、分批发送 | 订阅用户 > 1K |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| AI API Key 硬编码在前端或客户端 | API Key 泄露，被恶意使用产生高额费用 | 所有 AI 调用必须通过后端代理，API Key 仅存于服务端环境变量 |
| 不限制用户输入到 AI 的内容 | Prompt injection 攻击，用户通过订阅话题注入恶意 prompt | 对用户输入进行严格的 sanitization，使用 system/user prompt 分离 |
| 爬虫存储了个人数据 | GDPR 违规，数据泄露风险 | 爬虫管道自动过滤个人数据（用户名、邮箱、IP） |
| 推送通知包含敏感信息 | 锁屏状态下信息泄露 | 推送通知仅显示"您有新的更新"，详细内容在应用内查看 |
| 用户订阅偏好未加密 | 用户兴趣画像泄露 | 敏感偏好数据加密存储 |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| 首次打开空白页 | 用户不知道做什么，立即流失 | 提供"推荐话题"和"热门订阅"引导新用户 |
| 信息流无时间标记 | 用户不知道内容的新鲜度 | 每条信息显示相对时间（"2 小时前"）和来源 |
| 摘要质量参差不齐 | 用户对 AI 摘要失去信任 | 实施摘要质量评分、用户反馈机制、低质量摘要标记 |
| 订阅管理复杂 | 用户无法有效管理订阅 | 简洁的订阅列表、一键退订、智能推荐相关话题 |
| 无离线支持 | 无网络时应用完全不可用 | 缓存最近的摘要，支持离线阅读 |
| 加载状态不明确 | 用户不知道应用是否在工作 | 清晰的 loading 状态、骨架屏、进度提示 |

## "Looks Done But Isn't" Checklist

- [ ] **爬虫管道:** 仅在本地测试通过——未测试生产环境的反爬措施、频率限制、网络异常
- [ ] **去重系统:** 仅用 URL 去重——未处理近似重复、通讯社转载、标题变体
- [ ] **AI 摘要:** 仅测试了少量文章——未测试大批量处理的性能和成本
- [ ] **推送通知:** 仅实现了发送功能——未实现频率控制、用户偏好、静默时段
- [ ] **用户系统:** 仅实现了注册登录——未实现数据导出/删除（GDPR）、订阅管理、付费逻辑
- [ ] **feed 监控:** 仅处理了正常 feed——未处理 feed 失效、格式错误、URL 变更
- [ ] **内容过滤:** 未实施——可能导致应用商店审核被拒或法律风险
- [ ] **成本监控:** 未实施——可能在上线后几天内超出预算

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| AI API 成本失控 | MEDIUM | 立即实施 token 预算限制 → 降级到更便宜的模型 → 添加语义缓存 → 优化 prompt 长度 |
| 爬虫被全面封锁 | HIGH | 转向 RSS + API 数据源 → 停止对被封锁网站的爬取 → 与网站协商合作 |
| 推送通知疲劳 | LOW | 立即降低推送频率 → 添加频率控制逻辑 → 允许用户自定义推送偏好 |
| GDPR 合规问题 | HIGH | 立即停止收集个人数据 → 聘请法律咨询 → 实施数据删除功能 → 更新隐私政策 |
| 应用商店被拒 | MEDIUM | 根据审核反馈修改应用 → 增加独特功能 → 准备更详细的审核材料 → 重新提交 |
| 去重失败 | MEDIUM | 添加内容指纹列 → 实施 SimHash/MinHash → 建立事件聚类机制 |
| RSS 源大面积失效 | MEDIUM | 启用 feed 监控告警 → 切换到备用源 → 建立 feed 健康数据库 |

## Pitfall-to-Phase Mapping

| Pitfall | Prevention Phase | Verification |
|---------|------------------|--------------|
| AI API 成本失控 | Phase 1 | 日志中每个请求都有 token 消耗记录；有成本监控 dashboard |
| 内容去重失败 | Phase 1 | 测试数据集中近似重复率 < 5%；有事件聚类展示 |
| 爬虫被反爬封锁 | Phase 1 | 管道在生产环境运行 1 周无大面积封锁；RSS 覆盖率 > 80% |
| RSS 源可靠性衰减 | Phase 1 | 有 feed 健康监控 dashboard；feed 失效后 24 小时内告警 |
| 版权纠纷 | Phase 1 | AI 摘要不包含原文完整句子；有 DMCA 响应流程文档 |
| 推送通知疲劳 | Phase 2 | 每用户日均推送 < 5 条；推送打开率 > 15% |
| GDPR 合规 | Phase 2 | 有隐私政策；有数据删除功能；有 DPA 文档 |
| 应用商店审核被拒 | Phase 3 | 应用有独特 AI 功能；内容过滤机制就绪；审核材料完整 |
| 数据库设计缺陷 | Phase 1 | 信息流查询 < 200ms；支持事件聚类查询 |
| 免费增值转化 | Phase 2 | 清楚每个免费用户的成本；付费转化率基线已建立 |
| 热点数据采集合规 | Phase 1 | 仅接入官方 API 源；有访问日志；有 DMCA 响应流程；无个人信息存储 |
